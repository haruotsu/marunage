# PR-71 loop / daemon — t_wada test list

## A. internal/loop construction

- [x] N1. `New` fails with `ErrInvalidConfig` when `WithRegistry` is omitted.
- [x] N2. `New` fails with `ErrInvalidConfig` when `WithTaskRepo` is omitted.
- [x] N3. `New` fails with `ErrInvalidConfig` when `WithDispatcher` is omitted.
- [x] N4. `New` fails with `ErrInvalidConfig` when `WithRender` is omitted.
- [x] N5. `New` defaults clock to `time.Now`, auditor to `config.NopAuditor`,
       max-parallel to 1.

## B. RunOnce — orchestration

- [x] O1. RunOnce on empty registry calls dispatcher once with the configured
       MaxParallel and renders to the configured destination.
- [x] O2. RunOnce iterates every plugin in the registry and upserts the
       returned source.Task rows into the tasks table.
- [x] O3. RunOnce skips a source.Task whose (Source, ExternalID) duplicates a
       row already present (ErrDuplicateExternalID is swallowed).
- [x] O4. RunOnce calls `Sincer.Since(ctx, checkpoint)` for plugins that
       implement Sincer; checkpoint is the kv_state value
       `loop.checkpoint.<plugin-name>` if present, "" otherwise.
- [x] O5. RunOnce updates the `loop.checkpoint.<plugin-name>` kv_state value
       when the plugin's Since/List succeeds (RFC3339Nano clock).
- [x] O6. A plugin that returns an error during List/Since does NOT abort
       the rest of discovery; an audit "loop.discover.fail" entry records
       the failing plugin name + error.
- [x] O7. RunOnce records `loop.start` and `loop.end` audit events bracketing
       the iteration.
- [x] O8. RunOnce returns early with ctx.Err() when ctx is cancelled before
       the first plugin runs.

## C. Run — interval loop

- [x] R1. `Run(ctx, interval)` calls RunOnce once immediately on entry, then
       again every `interval`, and returns nil when ctx is cancelled.
- [x] R3. A RunOnce that returns an error does NOT stop the loop; the next
       tick still runs. Every error gets an audit "loop.tick.fail" entry.
- [x] R4. Run with a non-positive interval returns ErrInvalidInterval
       without entering the loop.

## D. Lock-key safety

- [x] L1. With `WithLockKey` set, two concurrent Loops sharing the same
       kv_state see exactly one RunOnce execute and the other returns
       ErrLockBusy without dispatch / discover side effects.
- [x] L2. The lock is released when RunOnce returns so a later RunOnce can
       re-acquire it.
- [x] L3. A RunOnce panic still releases the lock (defer ReleaseLock).
- [x] L4. Without `WithLockKey`, concurrent RunOnce calls both proceed.

## E. CLI wiring

- [x] CLI1. `marunage loop --once` runs one iteration and exits 0.
- [x] CLI2. `marunage loop --interval 250ms` honours the interval and
       exits cleanly when the parent ctx is cancelled.
- [x] CLI3. `marunage loop` without --once or --interval defaults to
       `discovery.interval` from config (10m).
- [x] CLI4. `marunage daemon start` writes the pid via the daemonControl
       seam and prints it; --interval forwards through to `marunage loop`.
- [x] CLI5. `marunage daemon status` reports running / stale / no-pidfile
       distinctly so the operator knows whether to clean up.
- [x] CLI6. `marunage daemon stop` calls Stop with the documented timeout
       and propagates errors verbatim.
- [x] CLI7. `marunage daemon start` against an already-live daemon refuses
       with a non-zero exit and a message naming the existing pid.

## Cross-cutting

- [x] All existing tests still pass (`make test`).
- [x] `go test -race ./...` clean.
- [x] `make lint` green (no new findings).
