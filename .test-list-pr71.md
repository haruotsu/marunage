# PR-71 loop / daemon — t_wada test list

## A. internal/loop construction

- [ ] N1. `New` fails with `ErrInvalidConfig` when `WithRegistry` is omitted.
- [ ] N2. `New` fails with `ErrInvalidConfig` when `WithTaskRepo` is omitted.
- [ ] N3. `New` fails with `ErrInvalidConfig` when `WithDispatcher` is omitted.
- [ ] N4. `New` fails with `ErrInvalidConfig` when `WithRender` is omitted.
- [ ] N5. `New` defaults clock to `time.Now`, auditor to `config.NopAuditor`, max-parallel to 1.

## B. RunOnce — orchestration

- [ ] O1. RunOnce on empty registry calls dispatcher once with the configured
       MaxParallel and renders to the configured destination.
- [ ] O2. RunOnce iterates every plugin in the registry and upserts the
       returned source.Task rows into the tasks table (Insert with Source +
       ExternalID + Title + Body + Notes, status defaults to pending).
- [ ] O3. RunOnce skips a source.Task whose (Source, ExternalID) duplicates a
       row already present (ErrDuplicateExternalID is swallowed; no error).
- [ ] O4. RunOnce calls `Sincer.Since(ctx, checkpoint)` for plugins that
       implement Sincer; checkpoint is the kv_state value
       `loop.checkpoint.<plugin-name>` if present, "" otherwise.
- [ ] O5. RunOnce updates the `loop.checkpoint.<plugin-name>` kv_state value
       when the plugin's Since/List succeeds. Value is the current clock as
       RFC3339Nano.
- [ ] O6. A plugin that returns an error during List/Since does NOT abort
       the rest of discovery: every other plugin still runs, the dispatch
       phase still runs, and an audit "loop.discover.fail" entry records
       the failing plugin name + error.
- [ ] O7. RunOnce records `loop.start` and `loop.end` audit events bracketing
       the iteration.
- [ ] O8. RunOnce returns early with ctx.Err() when ctx is cancelled before
       the first plugin runs (cancellation honoured).

## C. Run — interval loop

- [ ] R1. `Run(ctx, interval)` calls RunOnce once immediately on entry,
       then again every `interval`, and returns nil when ctx is cancelled.
- [ ] R2. Cancellation between ticks unblocks the timer and returns nil
       (not deadline exceeded).
- [ ] R3. A RunOnce that returns an error (catastrophic, not per-plugin)
       does NOT stop the loop: the next tick still runs. Every error gets
       an audit "loop.tick.fail" entry.
- [ ] R4. Run with a non-positive interval returns ErrInvalidInterval
       without entering the loop.

## D. Lock-key safety

- [ ] L1. With `WithLockKey("loop")` set, two concurrent Loops sharing the
       same kv_state see exactly one RunOnce execute and the other returns
       ErrLockBusy without dispatch / discover side effects.
- [ ] L2. The lock is released when RunOnce returns (success path), so a
       later RunOnce can re-acquire it.
- [ ] L3. A RunOnce panic still releases the lock (defer ReleaseLock).
- [ ] L4. Without `WithLockKey`, no lock is acquired and concurrent
       RunOnce calls both proceed.

## E. CLI wiring

- [ ] CLI1. `marunage loop --once` runs one iteration and exits 0; output
       confirms "loop: one iteration complete" or similar.
- [ ] CLI2. `marunage loop --interval 100ms` honours the interval and exits
       cleanly when SIGTERM-style cancellation arrives. (Test by cancelling
       cmd.Context after a few ticks.)
- [ ] CLI3. `marunage loop` without --once or --interval defaults to
       interval = discovery.interval from config.
- [ ] CLI4. `marunage daemon start` writes a pidfile with the running
       process id and prints the pidfile path.
- [ ] CLI5. `marunage daemon status` reports "running" with the pid when
       the pidfile points to a live process; "stopped" otherwise.
- [ ] CLI6. `marunage daemon stop` sends SIGTERM to the pid, removes the
       pidfile after the process exits.
- [ ] CLI7. `marunage daemon start` refuses to start when an already-live
       daemon is detected (returns non-zero with a clear message).

## Cross-cutting

- [ ] `make test` (race detector clean) green.
- [ ] `make lint` green (no new findings).
