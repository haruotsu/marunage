# PR-43: atomic sentinel による完了検知

このworktreeは `feat/pr-43-atomic-sentinel` ブランチで作業する。base は `main` (82838bc)。

## 背景

PR-42 (PR #21, merged) で dispatcher が `pending → running` 遷移と ws 起動・プロンプト送信まで実装済み。完了検知 (`running → done`) は **本 PR-43 が初めて担当する**。

`docs/pr_split_plan.md` の "PR-43 atomic sentinel による完了検知" セクションが正本。

## 実装スコープ

1. **`.exit_code.tmp` → `mv .exit_code` パターン** — Claude セッション終了時にワークスペース内 cwd で `echo $? > .exit_code.tmp && mv .exit_code.tmp .exit_code` を実行させる。`mv` は同一ファイルシステム内で atomic なので、reader が "中途半端な exit_code を読む" race を排除。
   - 起動プロンプトの末尾 (PR-42 prompt 配線) で sentinel を書くよう指示する形になる可能性が高い (`internal/dispatch/prompt.go` の Base セクションに sentinel 書込指示を追加)
2. **ポーリング → `status='done'`** — sentinel ファイルの存在をポーリング (周期は config 経由で可変、デフォルト 5s 程度) し、検出したら `store.UpdateStatus(id, 'done')` + `result_summary` を保存。
   - sentinel 未到達でワークスペースが消えていた場合は **PR-44 reaper** の責務 — 本 PR では running 維持で良い (失敗扱いしない)
3. **`result_summary` の保存** — Claude の最終出力 (例: 直前 N 行の標準出力 / もしくは sentinel ファイルに併記された `.result_summary`) を `tasks.result_summary` に書き込む。

## アーキテクチャ案

- 新規パッケージ `internal/completion/` (もしくは `internal/sentinel/`) を作るのが自然。Dispatcher と疎結合に保ち、独立した goroutine / loop / CLI コマンドから呼べる構造に。
- `Watcher.Run(ctx)` のような形で running 状態のタスクを `store.List(Statuses=[running])` で引き、各 ws の sentinel を読みに行くポーリング loop。
- Watcher は `cmux.Client` (PR-40) を使って `cmux exec` 相当でファイル読み取り、もしくは workspace 内 cwd を直接 ls する方式 (cmux ws の実体がローカル fs にあるならファイル read で OK)。
- functional options で `WithStore` / `WithCmux` / `WithClock` / `WithPollInterval` を inject 可能にする。

**実装の選択肢が複数ある場合は `internal/dispatch/dispatch.go` のパターンに合わせて決める。** 推測で決めず、既存実装を読んで判断。
