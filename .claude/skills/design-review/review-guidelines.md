# 設計レビュー共通ガイドライン（marunage）

サブエージェントが共通で参照するレビュー観点。
各エージェントは自身の専門領域に加え、関連する共通観点もチェックすること。

> 本ガイドラインは `REQUIREMENTS.md` を「最上位の判断基準（SSOT）」として扱う。
> 設計判断が要件と矛盾する場合、要件側に明示的な改訂提案を出すこと。
> 矛盾を黙って見逃さない。**marunage は明示的非ゴールと既定セキュリティ条件が多い OSS** であり、判断の一貫性が価値の中核。

---

## 1. ドキュメント構造の完全性

設計ドキュメントには以下が含まれているか確認する：

- [ ] **概要 / 目的**: 何のための設計か 1〜3 行で言える
- [ ] **背景**: なぜ今これが必要か（既存課題・要件参照）
- [ ] **対象範囲・非対象**: スコープが明示されている
- [ ] **設計**: 採用案と代替案、選んだ理由
- [ ] **データモデル / I/F**: 必要に応じて schema・関数シグネチャ・JSON 例
- [ ] **段階的実装計画（Phase）**: 1 PR で収まる粒度の分割
- [ ] **リスク・トレードオフ**: 影響範囲と緩和策
- [ ] **テスト戦略**: 何をどのレイヤーで担保するか
- [ ] **ロールバック / 撤退条件**
- [ ] **オープンクエスチョン / 決定保留事項**

## 2. REQUIREMENTS.md との整合性（最重要）

設計が以下に矛盾していないか必ず確認する。矛盾があれば 🔴 重大として報告。

### 2.1 採用された 3.x 思想（採用必須）
- 3.1 対話型セッション + send 方式
- 3.2 ワークスペース永続化（完了後も削除しない）
- 3.3 自律性と人間の介入の両立
- 3.6 ローカル PC 常駐運用が第一級ターゲット
- 3.7 能動的に通知してくるエージェント（HEARTBEAT）
- 3.8 Markdown が SSOT、Git 管理、Obsidian Vault 互換
- 3.9 モデル・ランタイム中立
- 3.10 セキュリティを「思想」に格上げ
- 3.11 連携の摩擦をゼロに近づける

### 2.2 明示的「非ゴール」（採用禁止）
- 双方向の対話 Channel（Slack/Discord/Telegram から話しかける機能）を設けない
- マルチテナント / 組織管理機能（v0.2 時点）
- 共有サーバ / Kubernetes 運用最適化（v0.2 時点）
- 重量級ワークフロービルダー（n8n/Zapier 的 GUI）
- 中央集権的 Skill レジストリ（v0.3 まで）

設計が外部チャットアプリと「双方向対話」する経路を新設していないか必ず確認。

## 3. セキュリティ既定条件（§3.10 / §9.1）

以下が「既定 ON / 既定 OFF」として正しく設計されているか。

| 項目                                          | 既定 |
| --------------------------------------------- | ---- |
| API キー / OAuth トークンの平文保存           | 禁止 |
| Web UI / Gateway の `0.0.0.0` バインド        | 禁止 |
| サードパーティ Skill のサンドボックスなし実行 | 禁止 |
| サブスクリプション流用                        | 禁止 |
| OS Keychain / DPAPI / libsecret 連携          | 必須 |
| 構造化監査ログ（JSONL）                       | 必須 |
| 外部発信操作の人間承認モード                  | 必須 |
| `--dangerously-skip-permissions` の常用       | OFF  |
| アウトバウンド HTTP の domain allowlist       | 必須 |
| テレメトリ / 自動アップデート                 | OFF  |

**OpenClaw 反面教師**（§11.1）の各事象が再発しない設計になっているか必ず照合する。

## 4. プラガビリティ規約（§6）

新規アダプタ追加時、以下の 10 拡張点のどれに該当するかが明示されているか。

`Source / Store / Executor / Trigger / Notifier / Frontend / LLM Provider / Agent Runtime / Supervisor / Skill Registry`

実装形態は以下のいずれかが推奨：
- **シェルスクリプト**（stdin/stdout JSON）
- **Skill**（SKILL.md に自然言語）
- **HTTP**（adapter.toml）

## 5. Markdown SSOT / Obsidian Vault 互換（§3.8 / §5.13）

DB スキーマや CLI 設定を新設する設計の場合：

- [ ] Markdown を消したらタスク（または対象データ）が消えるか
- [ ] frontmatter で必要メタデータを表現できるか
- [ ] Vault ルートに特殊バイナリや隠しディレクトリを置いていないか（`.git` / `.obsidian` 以外）
- [ ] DB は SSOT ではなく検索インデックスとして従属しているか
- [ ] Obsidian Tasks プラグイン記法（`- [ ]` `- [x]`）と互換か

## 6. 常駐運用 / 静粛性（§3.6 / §5.10）

サービス・デーモン関連の設計の場合：

- [ ] launchd / systemd-user / タスクスケジューラのいずれかへ登録できるか
- [ ] アイドル時の CPU / ディスク / ネットワーク負荷がほぼ 0 になるか
- [ ] スリープ / 復帰イベントを購読し、差分回収できるか
- [ ] PID / lock ファイルでシングルトン保証されているか
- [ ] 自己復旧（指数バックオフ）が設計されているか
- [ ] `marunage doctor` 相当の死活確認手段があるか

## 7. 観測性 / 介入可能性（§5.6 / §9）

- [ ] すべてのツール呼び出し・Skill 発火・Notifier 送信・Discovery が構造化ログに残るか
- [ ] 失敗・復旧の履歴がダッシュボードに出るか
- [ ] 実行中のタスクへ `marunage open #n` 相当でアタッチできるか
- [ ] `marunage panic` 相当のキルスイッチがあるか

## 8. 図表の適切性

- [ ] Mermaid を使用して図表が作成されているか
- [ ] フローチャートで日本語を使う場合、ダブルクォートで囲まれているか
- [ ] アーキテクチャ図 / シーケンス図 / ER 図が必要箇所に配置されているか

## 9. Phase / 実装計画

- [ ] Phase 粒度が 1 PR 程度か
- [ ] Phase 間の依存が Mermaid で図示されているか
- [ ] 各 Phase の完了条件が定義されているか
- [ ] Phase ごとにロールバック可能か
- [ ] テスト計画が含まれているか

## 10. テスト方針（t_wada TDD）

- [ ] テストリストが先に提示されているか
- [ ] 1 テストずつ Red → Green → Refactor のサイクルを想定しているか
- [ ] 統合 / E2E のスコープが明確か
- [ ] モック乱用になっていないか（境界の偽装で本物のバグを隠していないか）

## 11. ライセンス / OSS 衛生（§13）

- [ ] MIT 互換でない依存を引き込んでいないか
- [ ] Conventional Commits 命名で書ける単位か
- [ ] ベンダーロックイン（特定 SaaS 必須化）になっていないか

## 12. 用語の一貫性

要件書で定義された語を勝手に言い換えない：
- Workspace ＝ Git リポジトリ ＝ Obsidian Vault
- Discovery / Heartbeat / Dispatcher / Notifier / Trigger
- atomic sentinel / 対話型 + send 方式 / 二重ループ
