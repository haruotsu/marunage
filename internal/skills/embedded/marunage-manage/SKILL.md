# marunage-manage

ルールエンジン（依存・締切・cwd・重複の決定的足切り）を通過した **ready 候補**
だけを受け取り、「今このエージェントに着手させてよいか」「着手するなら何を先に
やるべきか」を判断する管理層 (manage) の司令系統。`marunage` のループから
バッチで呼び出され、候補ごとに **スコア**（実行順）と **defer 判定**（今じゃない）
を返す。

重要な前提: **ここでは候補を捨てない。** 「やるべきでない（広告・スコープ外・
重複・情報不足）」はルールと収集層の早期 triage が既に落としている。この skill が
決めるのは *いつやるか* だけ — すなわち ready 候補の **順序づけ** と、価値はあるが
今ではない候補の **defer 線引き** である。判断に迷ったら drop ではなく、ready の
低スコア、もしくは defer に倒す。

このファイルはユーザがカスタマイズすることを前提にしている。OSS が同梱する
バージョンは初期値であり、自分の働き方・チームの優先順位・着手ポリシーに
合わせて編集してよい。編集後 `marunage setup --skills --check-updates` で
OSS 同梱版との差分を確認できる。

## 入力

1 回の呼び出しで複数の候補がまとめて渡される（バッチ評価 — コスト制御のため
1 候補 1 呼び出しにはしない）。各候補は次のフィールドを持つ:

- `index` — このバッチ内での 0 始まりの位置。出力はこの index で対応づける
- `source` — `slack` / `gmail` / `github` / `markdown` などの収集種別
- `external_id` — ソース側の識別子
- `title` — 短い要約
- `body` — 本文
- `notes` — 依存・締切などの付随メタデータ（JSON のことがある）
- `priority` — ソース由来の優先度ヒント（自由形式）

## 判定ロジック

各候補について、次の観点で総合的にスコアと defer を決める。

1. 緊急度 — 締切が近い・相手が待っている・ブロッカーになっているものを高スコアに
2. 重要度 — 影響範囲が広い・後続タスクを解放する・約束済みのものを高スコアに
3. 着手適性 — 情報が揃っていて機械に丸投げしても安全に進むものを高スコアに
4. いま着手すべきか — 価値はあるが「今じゃない」もの（締切が遠い・他の混雑・
   低優先で待てる）は `defer = true` にする。defer は捨てるのではなく後回し

スコアは相対的な順序づけのための数値で、高いほど先に着手する。負値や 0 も
許容する。`defer = true` の候補のスコアは無視される。

判定理由 (`reason`) は人間が後から `marunage review` で監査できるように、
どの観点でその結論に至ったかを 1 文で記述する。秘匿情報（トークン・本文の
生引用）を理由に書かない。

## 出力フォーマット

候補ごとに 1 オブジェクトを持つ **JSON 配列** を stdout に出力する。配列の長さは
入力候補数と一致させ、`index` で対応づける。スキーマ:

```json
[
  {"index": 0, "score": 10.0, "defer": false, "reason": "<1文>"},
  {"index": 1, "score": 0,    "defer": true,  "reason": "<1文>"}
]
```

- `index` — 入力候補の位置。欠けや重複があってはならない
- `score` — 実行順を決める数値。高いほど先。`defer = true` のときは無視される
- `defer` — `true` なら「今じゃない」（後回し）。`false` なら今着手してよい ready
- `reason` — 上記「判定ロジック」で適用した観点と根拠を 1 文で

出力例:

```json
[
  {"index": 0, "score": 100.0, "defer": false, "reason": "締切が3時間後で相手が待っている"},
  {"index": 1, "score": 5.0,   "defer": false, "reason": "影響は小さいが情報は揃っており安全に着手できる"},
  {"index": 2, "score": 0,     "defer": true,  "reason": "価値はあるが締切が2週間先で今は他が混雑している"}
]
```

JSON 配列だけを出力する。前置きや説明文を付けない。判断に迷う候補も必ず
score か defer のいずれかに倒し、曖昧な行や入力数と異なる長さの配列を出力しない。
パースできない出力は管理層が安全側（決定的スタブスコア）にフォールバックする。
