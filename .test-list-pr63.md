# PR-63 Web UI dashboard — t_wada test list

> Drive Red → Green → Refactor in this order. Tick each entry when the
> matching Go test goes green.

## A. internal/web Dashboard provider (snapshot aggregation)

- [ ] D1. `Snapshot` returns running tasks (id, source, title, ws, started_at, output preview), ordered by started_at ASC, id ASC.
- [ ] D2. `Snapshot` truncates the running output preview to the configured byte budget.
- [ ] D3. `Snapshot` returns the top-N pending tasks ordered by priority DESC, created_at ASC, id ASC.
- [ ] D4. `Snapshot` returns the total pending count distinct from the top-N slice.
- [ ] D5. `Snapshot` reports 24h totals: done / failed / skipped within `now - window`.
- [ ] D6. `Snapshot` reports per-source 24h breakdown sorted by source name.
- [ ] D7. `Snapshot` reports each registered source's auth status and last-listed checkpoint time.
- [ ] D8. `Snapshot` records `unknown` auth status when a plugin returns an error.
- [ ] D9. `Snapshot` stamps `GeneratedAt` from the injected clock.
- [ ] D10. `noopDashboardProvider` returns an empty snapshot with `GeneratedAt` set so handlers never deref nil.

## B. SQL-backed DashboardStore (production wiring)

- [ ] S1. `dashboardSQLStore.Running` selects status='running' rows ordered started_at ASC.
- [ ] S2. `dashboardSQLStore.PendingTop` selects pending rows by (priority DESC, created_at ASC, id ASC) and respects limit.
- [ ] S3. `dashboardSQLStore.PendingCount` returns the total pending count regardless of limit.
- [ ] S4. `dashboardSQLStore.Recent` aggregates done/failed/skipped per source within the window.
- [ ] S5. `dashboardSQLStore.SourceCheckpoints` returns the most-recent updated_at per `<source>_*` key.

## C. Handlers / templates / partials

- [ ] H1. `GET /` renders running, pending (count + top-N), 24h summary, and source status.
- [ ] H2. `GET /partials/dashboard` renders the dashboard fragment (no `<html>` wrapper).
- [ ] H3. `/partials/dashboard` sets `Cache-Control: no-store` so polling never reads stale browser cache.
- [ ] H4. The full page links the partials polling script (Content-Security-Policy already allows `script-src 'self'`).
- [ ] H5. Existing security-header / CSRF / SSE wire-up tests still pass.
